# 核心需求与模块边界 (Core Requirements & Boundaries)

本文档明确了 Rust 微内核系统的核心需求以及各模块的职责边界，作为系统设计与代码实现的准则。

## 1. 核心需求

系统需提供以下核心基础能力，使得上层业务（应用与插件）能够专注于领域逻辑：

*   **静态环境绑定 (Static Environment Binding)**：通过泛型环境约束（Environment Trait）在编译期静态决议各类组件及配置，实现绝对的内存安全与零成本抽象分离。
*   **配置热重载 (Hot Config Reloading)**：系统配置发生变更时，无需重启整个进程即可通知并应用到相关的模块。
*   **统一启停控制 (Unified Lifecycle Control)**：确保所有的应用和插件都有一致的启动、暂停、停止和资源释放机制，以支持平滑启停（Graceful Shutdown）。
*   **中间件编排 (Middleware Orchestration)**：提供拦截器/中间件层，允许在进入具体应用逻辑前进行日志记录、链路追踪、权限校验等横切处理。

## 2. 模块边界

*   **Host Proxy (系统本体)**：
    *   **职责**：作为微内核的载体，负责加载配置、扫描插件、执行生命周期代理和调度钩子。
    *   **限制**：不应包含任何具体的业务逻辑（如数据库查询、用户鉴权等）。
*   **Generic Apps (通用库/应用)**：
    *   **职责**：提供可复用的基础设施能力（例如数据库连接池、Redis缓存客户端等）。必须实现内核约定的全套生命周期钩子（创建前、创建后、挂载前、挂载后、重载、停止等）。
    *   **限制**：通过提供默认配置接口，允许 Host 本体在未配置时自动覆盖使用默认参数，做到开箱即用。
*   **Business Plugins (业务插件)**：
    *   **职责**：实现具体业务（如用户管理、权限管理），向内核注册自身的实例或暴露服务接口，与其他插件进行通信。
    *   **限制**：依赖的外部资源必须通过环境泛型静态约束获取，自身需处理资源的正确释放（Destroy 阶段）。
